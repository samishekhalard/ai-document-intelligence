# Architecture

This document describes the platform's architecture in depth — the data
flow, the module boundaries, and the reasoning behind each non-obvious
choice.

## 1. Goals and Non-Goals

### Goals

- **Local-first**: run end-to-end on a developer laptop without any paid
  third-party service.
- **Grounded answers**: every generated answer should be traceable to
  source chunks.
- **Multilingual**: support at least English + Russian out of the box
  with no model-swapping.
- **Observable**: query latency, confidence, and agent traces must be
  inspectable from the UI and the API.
- **Deterministic tests**: the unit suite must run in under a second
  without Ollama, without network, and without flakiness.

### Non-goals (for v1.0)

- Distributed / multi-node deployment.
- Streaming token responses.
- Fine-tuned cross-encoder reranking.
- Role-based access control.

## 3. Request Lifecycle

### 3.1 Ingestion — `POST /upload`

```
client ─► FastAPI ─► IngestionService
                        │
                        ├─► IngestionPipeline.prepare
                        │     │
                        │     ├─ MIME sniffing and format dispatch
                        │     ├─ pypdf / python-docx / tesseract fallback
                        │     ├─ Recursive character splitter (512/64)
                        │     ├─ langdetect → language metadata
                        │     └─ keyword classifier → doc_type metadata
                        │
                        ├─► EmbeddingPipeline.add_chunks
                        │     ├─ Ollama /api/embeddings (nomic-embed-text)
                        │     └─ ChromaDB upsert (cosine, HNSW)
                        │
                        └─► MetadataStore.upsert_document
```

Failure modes:

- File not readable → 404.
- No extractable text → 422.
- Embedding backend unreachable → 500 with structured body.

### 3.2 Query — `POST /query`

```
client ─► FastAPI ─► OrchestratorAgent.run
                        │
                        ├─► router: inspect query + image_path
                        │     ├─ if image: → vision
                        │     ├─ if "summarise / entities / ...": → nlp
                        │     └─ default: → rag
                        │
                        ├─► rag:
                        │     ├─ RetrievalPipeline.retrieve
                        │     │     ├─ dense top-20 (Chroma)
                        │     │     ├─ sparse top-20 (BM25)
                        │     │     ├─ min-max normalise + weighted fuse
                        │     │     └─ token-overlap rerank → top-5
                        │     ├─ Format context with [1]..[N] citations
                        │     └─ Ollama chat (llama3.2)
                        │
                        ├─► (fallback) nlp if RAG confidence < 0.15
                        │
                        └─► finalise: assemble QueryResponse
```

The orchestrator is capped at `AGENT_MAX_ITERATIONS=3` (configurable).
Iteration count increments on every router pass so a graph that would
loop forever terminates.

### 3.3 NLP analysis — `POST /analyze`

Bypasses the orchestrator; directly invokes the `NLPAgent`. Supports
text input or `document_id` referring to an indexed document (whose
chunks are concatenated before analysis).

### 3.4 OCR — `POST /extract-ocr`

Bypasses the orchestrator; reads bytes, preprocesses with Pillow,
invokes `pytesseract` with configurable language string.

## 4. Retrieval Pipeline in Detail

### 4.1 Dense retrieval

Embeddings are generated by `nomic-embed-text` (768-dim) via Ollama and
stored in ChromaDB with `hnsw:space=cosine`. The pipeline pre-fetches
`max(RETRIEVAL_TOP_K, top_k*4)` candidates to give the reranker room to
reshuffle.

### 4.2 Sparse retrieval

A per-process BM25 index is rebuilt on the fly whenever the vector
store's count changes. This keeps the sparse index in sync with
Chroma without a second persistence layer. For corpora above ~50k
chunks, the rebuild cost dominates and a persistent BM25 implementation
should be substituted (e.g. `whoosh`, `tantivy`).

### 4.3 Fusion

Scores are min-max normalised into [0, 1] and combined as

```
fused = DENSE_WEIGHT * dense_norm + SPARSE_WEIGHT * sparse_norm
```

Defaults: 0.7 dense, 0.3 sparse. Tune via `.env`.

### 4.4 Reranking

The fused top-20 is reranked by `0.7 * fused + 0.3 * token_overlap`,
where `token_overlap = |query_tokens ∩ chunk_tokens| / |query_tokens|`.
Final top-5 is returned.

This "poor man's cross-encoder" recovers most of the lift a real
cross-encoder gives on enterprise corpora without downloading a 500 MB
HuggingFace checkpoint.

## 5. Storage

| Store | Purpose | File |
|---|---|---|
| ChromaDB persistent | Vectors + payload + metadata | `data/chroma/` |
| SQLite | Document metadata, query metrics | `data/metadata.db` |
| Filesystem | Raw uploaded files | `data/uploads/` |
| Log file | Rotating loguru sink | `logs/app.log` |

All paths are configurable via `.env`.

## 6. Concurrency Model

FastAPI + Uvicorn run a single-process, async event loop. Heavy CPU
operations (chunking, embedding RPC, tesseract) run synchronously on
the event loop thread; this is acceptable for a single-node deployment
and simplifies debugging. For horizontal scaling, move CPU-bound work
to a worker pool (e.g. `asyncio.to_thread`) or spawn Uvicorn workers.

SQLite access is serialised by a process-level `threading.Lock` in
`MetadataStore` to avoid "database is locked" errors under concurrent
ingestion.

## 7. Configuration

All tunable parameters are sourced from environment variables via
`pydantic-settings`:

- `OLLAMA_BASE_URL`, `OLLAMA_LLM_MODEL`, `OLLAMA_EMBED_MODEL`
- `CHUNK_SIZE`, `CHUNK_OVERLAP`
- `DENSE_WEIGHT`, `SPARSE_WEIGHT`
- `RETRIEVAL_TOP_K`, `RERANK_TOP_K`
- `AGENT_MAX_ITERATIONS`

See `.env.example` for the complete annotated list.

## 8. Testing Strategy

- **Unit tests**: fast, deterministic, no external services. Monkey-patch
  agent methods to avoid LLM calls.
- **Integration tests** (future): run against a Docker-compose stack
  with Ollama and hit the real `/query` path.
- **Evaluation harness**: 20 Q&A pairs exercising the end-to-end
  retrieval + generation path, producing precision@5 and latency
  metrics.

## 9. Extension Points

- **Swap the vector store**: `EmbeddingPipeline._chroma` is the only
  ChromaDB reference. Replace with a Qdrant / pgvector client and
  adapt `add_chunks` and `collection.query`.
- **Swap the LLM**: replace `ollama.Client` in `RAGAgent.__init__` with
  any chat-like client — the prompt contract is stable.
- **Add agents**: extend `OrchestratorAgent` with a new node and
  conditional edge; update `OrchestratorState` with the new response
  field.

## 10. Security Posture

- Inputs are parsed by structured libraries (`pypdf`, `python-docx`,
  `PIL`); no `subprocess` or `eval` on user-supplied data.
- OCR is run on files the service itself wrote to disk; the path is
  not attacker-controlled.
- CORS is wide-open for local development. Production overrides are
  documented in `docs/deployment.md`.
- No secrets are read from request bodies; environment variables are
  the only secret source.
